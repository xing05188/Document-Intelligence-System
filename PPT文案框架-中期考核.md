## 一、项目概述

### 1.1 项目背景

| 条目 | 内容 |
|------|------|
| **项目名称** | 基于大语言模型的文档理解与多源数据融合系统（Document Intelligence System） |
| **项目定位** | 面向企业和个人的智能文档处理平台，融合LLM能力实现文档的理解、编辑、数据提取与自动化工作流编排 |
| **行业背景** | 企业日常运营中产生大量非结构化文档（年报、合同、研究报告等），人工处理效率低、易出错；传统OCR和规则引擎难以应对复杂语义理解需求 |
| **技术背景** | 大语言模型（LLM）在文本理解、生成和推理方面取得突破性进展，为文档智能化处理提供了新的技术路径 |

### 1.2 项目目标

- **核心目标**：构建一套基于LLM的文档智能处理系统，实现从文档输入、理解、编辑、数据提取到结构化存储的全链路自动化
- **具体目标**：
  - 支持多格式文档（DOCX、PDF、TXT、MD、XLSX）的解析与理解
  - 实现自然语言驱动的文档编辑（格式调整、内容修改等）
  - 实现非结构化文档中实体数据的智能提取与结构化输出
  - 实现Excel数据的智能筛选与模板填表
  - 提供可视化工作流编排能力，支持用户自定义文档处理流水线
  - 支持多种LLM后端接入（DeepSeek、智谱等）

### 1.3 项目意义

| 维度 | 说明 |
|------|------|
| **应用价值** | 大幅提升文档处理效率，降低人工成本；支持批量处理，适用于年报分析、合同审查、数据录入等高频场景 |
| **技术价值** | 探索LLM在文档智能领域的工程化落地路径；验证多Agent协作架构在复杂文档任务中的有效性 |
| **学术价值** | 为文档理解、信息抽取、工作流编排等方向提供可复用的系统设计与实践经验 |

- 列表处理：项目符号列表、编号列表
- 特殊元素：超链接插入

### 2.3 模块二：实体提取 Agent（Agent B）

**功能概述**：从非结构化文档（DOCX、PDF、TXT、MD）中智能提取结构化数据，输出为JSON格式。

**核心能力**：
- 支持自然语言描述提取需求
- 支持Excel模板列名引导提取字段
- 大文本智能分块处理，突破Token限制
- 多线程并发提取，提升处理效率
- 输出结构化JSON，包含值、置信度、证据溯源

**工作流程**：
```
输入文档 → 文本解析 → 模板列识别 → LLM分块提取 → JSON结果聚合 → 输出
```

### 2.4 模块三：数据库存储 Agent（Agent C）

**功能概述**：将Agent B提取的结构化数据持久化存储到PostgreSQL数据库。

**核心能力**：
- 支持PostgreSQL / Supabase接入
- JSONB格式存储提取结果，支持版本管理
- 任务状态机管理（queued → running → succeeded/failed）
- 审计日志记录关键操作事件
- 提供RESTful API查询接口

| audit_logs | 审计日志 |
| document_assets | 文档元数据 |
| fill_reports | 填表报告 |
| agent_execution_logs | Agent执行日志 |

### 2.5 模块四：Excel数据筛选与填表 Agent（Agent D）

**功能概述**：从Excel文件中根据自然语言条件筛选数据行，并支持将结果填入Word/Excel模板。

**核心能力**：
- 自然语言筛选条件解析（如"筛选出销售额大于100万的记录"）
- 14种筛选算子支持：等于、不等于、大于、小于、包含、介于、正则匹配等
- 支持多条件组合筛选（AND/OR）
- 筛选结果按字段映射填入Word模板（邮件合并风格）
- 支持多表填表（Word表格 + Excel工作表）

### 3.3 关键技术方案

#### 3.3.1 多Agent协作架构

采用**编排-执行**分离模式，WorkflowCoordinator作为中央调度器，根据TaskType路由到对应的Agent执行：

```
TaskSpec → WorkflowCoordinator → 匹配Handler → Agent执行 → 结果聚合 → 输出
```

- **TaskSpec**：统一的任务规格定义，包含任务类型、指令、源文件、参数等
- **WorkflowCoordinator**：负责任务路由、生命周期管理、进度回调
- **各Agent独立部署/调用**：通过统一的BaseAgent接口规范，便于扩展

#### 3.3.2 LLM集成方案

- **多Provider抽象**：通过LLMService封装不同LLM提供商的API差异
- **已接入Provider**：DeepSeek（deepseek-chat）、智谱AI
- **流式输出**：支持SSE（Server-Sent Events）实时推送LLM回复
- **可配置参数**：temperature、max_tokens、streaming等

#### 3.3.3 工作流引擎设计

- **可视化编排**：基于Vue 3实现的拖拽式画布，节点自由排列与连线
- **节点Schema驱动**：每个节点类型对应一套Schema定义，动态生成配置面板
- **Pipeline执行**：按节点顺序依次执行，中间产物在内存/临时文件传递
- **状态持久化**：执行状态实时写入JSON文件，支持断点恢复

#### 3.3.4 文档编辑引擎（Agent A）

- **指令解析流水线**：自然语言 → LLM解析 → ActionPlan（结构化动作序列）→ 逐动作执行
- **能力矩阵校验**：执行前校验动作与文件类型的兼容性
- **降级策略**：核心功能实现两层降级（样式名称 → XML原生操作）
- **多Adapter模式**：DOCX / MD / TXT / XLSX各有一套Adapter实现

#### 3.3.5 实体提取方案（Agent B）

- **语义分块**：按语义边界切分大文本，确保每个分块语义完整
- **多线程并发**：ThreadPoolExecutor并发调用LLM提取，大幅提升效率
- **模板引导**：支持Excel模板列名作为提取字段的引导
- **结果聚合**：多分块提取结果去重合并，形成完整JSON输出

#### 3.3.6 数据筛选引擎（Agent D）

- **算子体系**：14种筛选算子，覆盖常见比较、包含、正则等操作
- **LLM条件解析**：自然语言筛选条件 → 结构化筛选计划
- **多表填表**：支持Word模板（邮件合并）和Excel模板两种填表模式

### 3.4 数据库设计

**核心表关系**：

```
tasks (任务主表)
  ├── task_id (业务键, TEXT UNIQUE)
  ├── status (queued/running/succeeded/failed/review/cancelled)
  ├── task_type
  └── metadata (JSONB)
       │
       ├── extraction_results (抽取结果)
       │     ├── task_uuid → tasks.id
       │     ├── payload (JSONB) ← Agent B输出
       │     └── result_version (版本号)
       │
       ├── audit_logs (审计日志)
       │     ├── subject_type/subject_id
       │     └── event/actor/changes (JSONB)
       │
       ├── document_assets (文档元数据)
       │     ├── file_name/file_type/file_size
       │     └── storage_key (对象存储路径)
       │
       ├── fill_reports (填表报告)
       │     └── report_data (JSONB) ← Agent D输出
       │
       └── agent_execution_logs (Agent执行日志)
             └── log_data (JSONB) ← Agent A输出
```

| **画布连线退化** | 二次贝塞尔控制点与端点共线，连线退化为僵硬水平线 | 改用正交折线+三次贝塞尔混合策略，根据纵向差动态选择连线类型 |
| **文档库数量不一致** | 工作流入库后左侧数字变了但网格列表未更新 | loadSpaces成功后清空docsCache并强制刷新当前空间文档列表 |
| **PDF依赖缺失** | 工作流输出PDF时报错"No module named 'reportlab'" | requirements.txt补充reportlab依赖 |
| **文档库入库user_id** | 入库记录user_id为空导致已登录用户看不到入库文件 | 入库时使用document_spaces的所属用户ID |
| **大文本Token超限** | 长文档超出LLM上下文窗口限制 | 实现语义分块+多线程并发提取+结果聚合策略 |
| **项目符号/编号兼容性** | 部分DOCX模板样式名称不标准导致列表格式失败 | 实现两层降级策略：优先样式名称，失败后使用XML原生操作 |
